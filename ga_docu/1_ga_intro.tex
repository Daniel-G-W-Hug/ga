\subsection{Introduction}
\label{intro}

The ga library is intended to make Geometric Algebra (GA) more accessible by providing
functionality for numerical calculations required for simulation of physical systems. \\

It implements functionality for two- and three-dimensional spaces with Euclidean geometry.
Currently, it provides data types and operations to handle the algebras of Euclidean
geometric algebra $(ega2d, ega3d)$ and projective Euclidean geometric algebra $(pga2dp,
pga3dp)$. The library efficiently handles fully populated multivectors, even and odd grade
multivectors as well as their components like scalars, vectors, bivectors, trivectors, and
pseudoscalars of the corresponding linear spaces. It also provides operations to work with
these objects. It was designed with easy extensibility in mind and can be extended to
provide functionality for handling additional algebras, e.g. spacetime algebra $(STA)$,
with limited effort. \\

The main library is a header only library in the \path{ga} sub-folder of the library which
can be
used by including either
\begin{verbatim}
    #include ga/ga_ega.hpp // and/or
    #include ga/ga_pga.hpp
\end{verbatim}
and by making the corresponding namespaces accessible by stating
\begin{verbatim}
    using namespace hd::ga;      // and
    using namespace hd::ga::ega; // either/or
    using namespace hd::ga::pga;
\end{verbatim}
in your application code. For a simple usage example please have a look either at
\path{ga/ga_test/src/ga_ega_test.cpp} or at \path{ga/ga_test/src/ga_pga_test.cpp}. \\

At the heart of the library there are user defined types that model mathematical objects
of the respective geometric algebra. They are realized using type tagging in order to
maximize reuse and to avoid code duplication. All types are defined in
\path{ga/detail/type_t}. \\

To give an example for $EGA2D$, there are type aliases defined in the file
\path{ga/detail/type_t/type2d.hpp} which provide types for scalars, vectors,
pseudoscalars, even grade multivectors and full multivectors with the corresponding number
of components. We have to distinguish between scalars and pseudoscalars at type level in
order to allow for complement operations (defined later on) that can distinguish those
types for each algebra. Following types are provided:
\begin{verbatim}
namespace hd::ga {

template <typename T> using Scalar2d
                           = Scalar_t<T,scalar2d_tag>;
template <typename T> using Vec2d = Vec2_t<T,vec2d_tag>;
template <typename T> using PScalar2d
                           = Scalar_t<T,pscalar2d_tag>;
template <typename T> using MVec2d_E = MVec2_t<T,mvec2d_e_tag>;
template <typename T> using MVec2d = MVec4_t<T,mvec2d_tag>;

} // namespace hd::ga
\end{verbatim}
Based on these template aliases types like \path{Vec2d<T>} can be used directly. However,
for increased user convenience additional template specializations have been defined in
\path{ga/ga_usr_types.hpp} as
\begin{verbatim}
using scalar2d = Scalar2d<value_t>;
using vec2d = Vec2d<value_t>;
using pscalar2d = PScalar2d<value_t>;
using mvec2d_e = MVec2d_E<value_t>;
using mvec2d = MVec2d<value_t>;
\end{verbatim}
where \path{value_t} is defined for the whole library in \path{ga/value_t.hpp}. Thus the
user can simply select \path{value_t = double} (set as default value) or \path{float} for
the library and the user application at one location. There is no need to use template
parameters at all, if the user-defined types like \path{scalar2, vec2d, pscalar2d,
mvec2d_e}, and \path{mvec2d} are used for the application. \\

All algebras follow the same pattern. Just to give a second example for $PGA3DP$ following
user types are defined: \path{scalar3dp, vec3dp, bivec3dp, trivec3dp, pscalar3dp,
mvec3dp_e, mvec3dp_u,} and \path{mvec3dp}, where we additionally have defined types for
bivectors, trivectors and uneven grade multivectors, as required to implement the algebra.
\\

To work with the user defined algebraic objects geometric algebra operations using these
types are defined in the ga library. You will find all operations that are required to
work with linear vector spaces and operations specific to geometric algebra.
\\

All product expressions for inner, outer, interior and geometric products as well as their
regressive counterparts are generated by programs in the subfolder \path{ga/ga_prdxpr}.
User input to configure product generation is provided in the header files of the
corresponding algebra as well. The generated coefficient expressions are used in the
source code to implement dot products, wedge products, commutator products, contractions,
expansions, geometric products or their regressive counterparts, as well as sandwich
products describing rotors and motors of the corresponding algebra. If certain products
are required for your applicaton, but not yet implemented in the ga library, they can be
easily generated within \path{ga/ga_prdxpr/src_prdxpr} as needed and added with minimal
effort. \\

As mentioned before, we distinguish inner, interior, outer and geometric products.
\begin{itemize}
    \item Inner ($a \cdot b$) and interior products, like contractions and expansions ($a
    \vee \rightbulkdual{b}$, $a \wedge \rightbulkdual{b}$), are grade reducing operations.
    The inner product combines arguments of the same type and grade, while the interior
    products involve duals of one of their arguments, combining and linking arguments of
    different grades. Inner products are linked to the metric of the modelled space. 
    \item The outer product ($a \wedge b$) is a grade-increasing operation adding the
    grades of its arguments. Outer products, like the wedge product, connect subspaces
    represented by their operands and create new subspaces defined by the span of the
    subspaces of the operands. They establish a join-operation between the involved
    subspaces.
    \item The geometric product ($a \wedgedot b$) is the combination of both types of
    products in one expression. They can result in multivectors consisting of parts with
    different grades as result of the operation. The main advantage of the geometric
    product is its invertibility under certain circumstances.
    \item Regressive products ($a \circ b \equiv \underline{\bar{a} \cdot \bar{b}}$, $a
    \vee b \equiv \underline{\bar{a} \wedge \bar{b}}$, $rcmt(a,b) \equiv
    \underline{cmt(\bar{a},\bar{b})}$, $a \veedot b \equiv \underline{\bar{a} \wedgedot
    \bar{b}}$) use complement operations on their arguments before and after applying the
    operation\footnote{for definition of the left and right complement operations marked
    with underbar and overbar, see next section}. The regressive wedge product creates an
    intersection of the (sub-)spaces used by their arguments, thus effectively creating a
    meet-operation between the involved spaces.
    \item Sandwich products ($R \wedgedot u \wedgedot \revtilde{R}$ for $EGA$, $M \veedot
    u \veedot \revtilde{M}$ for $PGA$) are used to implement orthogonal transformations of
    geometric objects. They realize rotations for $EGA$ and general motions including
    rotation and translation for $PGA$ by concatenating two consecutive reflections, used
    as primitives to realize the resulting transformations. This can be achieved
    algebraically by multiplying the object symmetrically from the left- and right-hand
    side with the appropriate unitized objects, we either call rotors $R$ in $EGA$ or
    motors $M$ in $PGA$.
\end{itemize}

Following bi-argument operations $\emph{op}(a,b) = a \; op \; b$ and their regressive
variants $\emph{rop}(a,b) = \underline{\bar{a} \; \emph{op} \; \bar{b}}$ are implemented
(with \path{op(a,b)} or \path{rop(a,b)} as the name of the implemented function). Using
$A$ vs. $a$ implies $gr(A) \ge gr(a)$, where $gr(arg)$ returns the grade of the argument
$arg$:
\begin{itemize}
    \item Inner product, delivering a scalar: $a \cdot b$, \path{dot(a,b)}
    \item Outer product of a $j$- and a $k$-vector, delivering a $j+k$-vector in
    $span(a,b)$: $a \wedge b$, \path{wdg(a,b)}
    \item Geometric product:  $a \wedgedot b$, \path{operator*(a,b)}
    \begin{itemize}
        \item For vectors $a$ and $b$:  $a \wedgedot b = a \cdot b + a \wedge b$
        \item For a vector $a$ and a multivector $B$:
        $a \wedgedot B = B \gg a + a \wedge B$
                \item For a multivector $A$ and a vector $b$:
        $A \wedgedot b = b \ll A + A \wedge b$
    \end{itemize}

    \item Commutator product (the asymmetric part of the geometric product): $cmt(a,b)
    \equiv \frac{1}{2}(a \wedgedot b - b \wedgedot a)$,
    \item Regressive commutator product (the asymmetric part of the regressive geometric
    product): \\
    $rcmt(a,b) \equiv \frac{1}{2}(\underline{\bar{a} \wedgedot \bar{b} - \bar{b}
    \wedgedot \bar{a}}) = \frac{1}{2}(a \veedot b - b \veedot a)$,
    \path{rcmt(a,b)}
    \item Regressive inner product, delivering a pseudoscalar: $a \circ b$,
    \path{rdot(a,b)}
    \item Regressive outer product: $a \vee b$, \path{rwdg(a,b)}
    \item Regressive geometric product: $a \veedot b$, \path{rgpr(a,b)}
    
    \item Left (bulk) contraction: $a \lcontr B = a \ll B = a_{\star} \vee B $,
    \path{operator<<(a,B)}, which implements \path{left_bulk_contract(a,B)}  
    \item Right (bulk) contraction: $B \rcontr a = B \gg a = B \vee a^{\star} $,
    \path{operator>>(B,a)}, which implements \path{right_bulk_contract(B,a)}

    \item Left weight contraction: $a_{\smallstar} \vee B $,
    \path{left_weight_contract(a,B)} 
    \item Right weight contraction: $B \vee a^{\smallstar} $,
    \path{right_weight_contract(B,a)} 

    \item Left bulk expansion: $A_{\star} \wedge b $,
    \path{left_bulk_expand(A,b)} 
    \item Right bulk expansion: $a \wedge B^{\star} $,
    \path{right_bulk_expand(a,B)} 

    \item Left weight expansion: $A_{\smallstar} \wedge b $,
    \path{left_weight_expand(A,b)} 
    \item Right bulk expansion: $a \wedge B^{\smallstar} $,
    \path{right_weight_expand(a,B)} 
\end{itemize}
where the items from the left contraction downwards are all considered interior products.
They operate directly on the subspaces occupied by the objects. The product operation is
performed after applying a dualization operation to one operand. \\

The product tables for all products are generated in \path{ga/ga_prdxpr}, and coefficient
expressions for specific argument types like scalars, vector, bivectors, etc. can be
generated based on those product tables using configuration options of
\path{ga/ga_prdxpr}. \\

$PGA$ explicitly means Projective Geometric Algebra in the context of the ga library.
There is also an alternative approach called Plane-based Geometric Algebra (also
abbreviated $PGA$) that is based on a space that is dual to the space modelled and used
here. The alternative approach was introduced by~\cite{Gunn_PGA:2020} and further promoted
for applications in projective geometry
by~\cite{Dorst_deKennick_A_guided_tour_to_PGA:2022}. That model uses dual representations
of primarily used geometric objects represented in the algebra. Planes are used instead of
vectors to model everything, e.g. a point is modelled by three planes intersecting in a
point. Both representations are mathematically equivalent, i.e. dual to each other. They
are linked by the definition of complements and the interconnection between geometric and
regressive geometric product, as defined above: $a \veedot b \equiv \underline{\bar{a}
\wedgedot \bar{b}}$. In the plane-based approach the geometric product is used to operate
on complements and duals, which are defined as the primary objects used for the approach,
while the approach introduced by Lengyel in~\cite{Lengyel_pga-illuminated:2024} favors the
use of points, vectors, bivectors and planes plus the explicit use of regressive products
and complements as well as duals for applying geometric operations to these objects. The
latter approach unifies the use of the wedge product for join operations and of the
regressive wedge product for meet operations for application in $EGA$ and $PGA$. \\
