\subsection{Introduction to the ga library}
\label{subsection:intro}

The ga library is intended to make Geometric Algebra (GA) more accessible by providing
functionality for numerical calculations required for simulation of physical systems. \\

It implements functionality for two- and three-dimensional spaces with Euclidean geometry.
Currently, it provides data types and operations to handle the algebras of two- and
three-dimensional Euclidean space ($EGA2D$, $EGA3D$) as well as for projective two- and
three-dimensional Euclidean space ($PGA2DP$, $PGA3DP$). The latter two are modelled in a
three- and four-dimensional space, respectively, since both include an additional
projective dimension. \\

For the ga library we use linear spaces $\mathbb{G}(p,n,z)$, where the arguments signify:
$p$ = number of basis vectors squaring to +1, $n$ = number of basis vectors squaring to
-1, and $z$ = number of basis vectors squaring to 0. The linear space $\mathbb{G}$ is an
extension of a linear vector space to a linear space of multivectors in the respective
basis (for details see \cite{Macdonald:2010}).
\begin{itemize}
\item $EGA2D$: $\mathbb{G}(2,0,0)$ (extends $\mathbb{R}^2$)
\item $EGA3D$: $\mathbb{G}(3,0,0)$ (extends $\mathbb{R}^3$)
\item $PGA2DP$: $\mathbb{G}(2,0,1)$
\item $PGA3DP$: $\mathbb{G}(3,0,1)$
\end{itemize}

The library efficiently handles fully populated multivectors, even- and odd-grade
multivectors as well as their components like scalars, vectors, bivectors, trivectors, and
pseudoscalars of the corresponding linear spaces. It also provides operations to work with
these objects. It was designed with extensibility in mind and could be extended to provide
functionality for handling additional algebras, e.g. spacetime-algebra $(STA)$, with
limited effort. \\

The core implementation language of the library is \Cpp. The documentation is focused on
the C++ implementation exclusively. However, the library provides wrappers for python
$(3.10+)$ in \path{ga_py} that is fully generated from the \Cpp implementation using
nanobind, i.e. it covers almost 100\% of \Cpp data types and functionality. There also is
a wrapper for lua in \path{ga_lua} (manually generated, with some remaining gaps in
coverage of \Cpp functionality).

\newpage

The main library is a header-only library in the \path{ga} sub-folder, which can be used
by including either
\begin{verbatim}
    #include ga/ga_ega.hpp // and/or
    #include ga/ga_pga.hpp
\end{verbatim}
and by making the corresponding namespaces accessible by stating
\begin{verbatim}
    using namespace hd::ga;      // and
    using namespace hd::ga::ega; // either/or
    using namespace hd::ga::pga;
\end{verbatim}
in your application code. For a simple usage example please have a look either at
\path{ga/ga_test/src/ga_ega_test.cpp} or at \path{ga/ga_test/src/ga_pga_test.cpp}. \\

\subsubsection{Data types -- user abstractions for each algebra}
\label{subsection:data_types}

At the heart of the library there are user defined types that model mathematical objects
of the respective geometric algebra. They are realized using type tagging in \Cpp to
maximize reuse and to avoid code duplication. All types are defined in
\path{ga/detail/type_t}. \\

To give an example for $EGA2D$, there are type aliases defined in the file
\path{ga/detail/type_t/type2d.hpp} which provide types for scalars, vectors,
pseudoscalars, even-grade multivectors and full multivectors with the corresponding number
of components. We have to distinguish between scalars and pseudoscalars at type level in
order to allow for complement operations (defined later on) that can distinguish those
types for each algebra. Following types are provided:
\begin{verbatim}
namespace hd::ga {

template <typename T> using Scalar2d
                           = Scalar_t<T,scalar2d_tag>;
template <typename T> using Vec2d = Vec2_t<T,vec2d_tag>;
template <typename T> using PScalar2d
                           = Scalar_t<T,pscalar2d_tag>;
template <typename T> using MVec2d_E = MVec2_t<T,mvec2d_e_tag>;
template <typename T> using MVec2d = MVec4_t<T,mvec2d_tag>;

} // namespace hd::ga
\end{verbatim}
Based on these template aliases types like \path{Vec2d<T>} can be used directly. However,
for increased user convenience additional template specializations have been defined in
\path{ga/ga_usr_types.hpp} as
\begin{verbatim}
using scalar2d = Scalar2d<value_t>;
using vec2d = Vec2d<value_t>;
using pscalar2d = PScalar2d<value_t>;
using mvec2d_e = MVec2d_E<value_t>;
using mvec2d = MVec2d<value_t>;
\end{verbatim}
where \path{value_t} is defined for the whole library in \path{ga/value_t.hpp}. Thus, the
user can simply select \path{value_t = double} (set as default value) or \path{float} for
the library and the user application at one location. There is no need to use template
parameters at all, if the user-defined types like \path{scalar2d, vec2d, pscalar2d,
mvec2d_e}, and \path{mvec2d} are used for the application. \\

All algebras follow the same pattern. Just to give a second example for $PGA3DP$ following
user types are defined: \path{scalar3dp}, \path{vec3dp}, \path{bivec3dp},
\path{trivec3dp}, \path{pscalar3dp}, \path{mvec3dp_e}, \path{mvec3dp_u}, and
\path{mvec3dp}, where we additionally have defined types for bivectors, trivectors and
uneven-grade multivectors, as required to completely implement the algebra. \\

To work with the user defined algebraic objects, the operations documented in the next
subsection using these types are defined in the ga library (and some more only documented
in source code). You will find operations required to work with linear (multi-)vector
spaces and operations specific to geometric algebra. In the following replace
\path{algebra} with any of \{\path{ega2d}, \path{ega3d}, \path{pga2dp}, \path{pga3dp}\}:
\begin{itemize}
\item \path{ga_algebra_ops_basics.hpp} contains basic operations like reversions,
conjugations, complements, duals,  norms, normalization and additionally for $PGA$ bulk,
weight and unitization operations.
\item \path{ga_algebra_ops_products.hpp} contains product operations like dot, wedge,
commutator, transwedge and geometric products, and their regressive counterparts, as well
as contractions, expansions, and inverses of blades.
\item \path{ga_algebra_ops.hpp} contains general operations like angle, exp, sqrt,
project\_onto, reflect\_from, is\_congruent, and sandwich products for motion operators
provided as rotate ($EGA$) and move2dp/3dp ($PGA$), as well as get\_rotor ($EGA$) and
get\_motor ($PGA$). The latter providing input arguments for the motion operations.
\item \path{ga_algebra_ops_physics.hpp} ($PGA$ only) contains basics for rigid body
dynamics calculations like inertia, its inverse and right hand sides for the dynamics
equations.
\end{itemize}



All product expressions for inner, outer, interior and geometric products as well as their
regressive counterparts are generated by programs in the subfolder \path{ga/ga_prdxpr}.
User input to configure product generation is provided in the header files of the
corresponding algebra. The generated coefficient expressions are used in the source code
to implement dot products, wedge products, commutator products, contractions, expansions,
geometric products or their regressive counterparts, as well as sandwich products
describing rotors and motors of the corresponding algebra. If certain products are
required for your application, but not yet implemented in the ga library, they can be
easily generated within \path{ga/ga_prdxpr/src_prdxpr} as needed and added with minimal
effort. \\

\subsubsection{Product types -- inner, interior, outer, geometric}
\label{subsection:product_types}

As mentioned before, we distinguish inner, interior, outer and geometric products.
\begin{itemize}
    \item Inner ($a \cdot b$) and interior products, like contractions and expansions ($a
    \vee \rightbulkdual{b}$, $a \wedge \rightbulkdual{b}$), are grade reducing operations.
    The inner product combines arguments of the same type and grade, while the interior
    products involve duals of one of their arguments, combining and linking arguments of
    different grades. Inner products are linked to the metric of the modelled space. They
    provide the capability of projecting objects like vectors on other objects like
    vectors or planes.

    \item The outer product ($a \wedge b$) is a grade-increasing operation adding the
    grades of its arguments. It extends the space that is taken by one object by the space
    that is taken by another object (e.g. two linear independent vectors, representing a
    linear subspace each, span a plane that is defined by the additive combination of
    those sub-spaces). The outer product (also called the wedge product) connects
    subspaces represented by their operands and creates new subspaces defined by the span
    of its subspaces of the operands. It establishes a join-operation between the involved
    subspaces.
    
    The outer product also allows for direct calculation of linear
    independence: if vectors a linear independent their outer product is zero, if they are
    linear dependent their outer product is unequal zero.

    \item The geometric product ($a \wedgedot b$) is the combination of both types of
    products in one expression. They can result in multivectors consisting of parts with
    different grades as result of the operation. The main advantage of the geometric
    product is that it is invertible under certain circumstances.

    \item Regressive products ($a \circ b \equiv \lcmpl{\rcmpl{a} \cdot \rcmpl{b}}$, $a
    \vee b \equiv \lcmpl{\rcmpl{a} \wedge \rcmpl{b}}$, $\mathrm{rcmt}(a,b) \equiv
    \lcmpl{\mathrm{cmt}(\rcmpl{a},\rcmpl{b})}$, $a \veedot b \equiv \lcmpl{\rcmpl{a}
    \wedgedot \rcmpl{b}}$) use complement operations on their arguments before and after
    applying the operation\footnote{For definition of the left and right complement
    operations marked with under-line and over-line, see next section.}. The regressive
    wedge product finds common lower-dimensional subspaces, i.e. it effectively finds the
    intersection of the sub-spaces represented by its arguments, thus effectively creating
    a meet-operation between the involved spaces. It can be used to find the intersection
    of sub-spaces, like the intersection point between a line and a plane, as a very
    simple product operation.

    \item Sandwich products ($R \wedgedot u \wedgedot \rev{R}$ for $EGA$, or $M \veedot u
    \veedot \rrev{M}$ for $PGA$) are used to implement orthogonal transformations of
    geometric objects $u$. They realize rotations for $EGA$ and general motions including
    rotation and translation for $PGA$ by concatenating two consecutive reflections to
    implement the resulting orthogonal transformations. In $2D$ the consecutive
    reflections are across lines, while in $3D$ the consecutive reflections are across
    planes. This can be achieved algebraically by multiplying the object symmetrically
    from the left- and right-hand side with the appropriate (typically unitized) objects,
    we either call rotors $R$ in $EGA$ or motors $M$ in $PGA$.
\end{itemize}

Following bi-argument operations \path{op(a, b)} $ =a \; \mathrm{op} \; b$ and their
regressive variants \path{rop(a, b)} $ = \lcmpl{\rcmpl{a} \; \mathrm{op} \; \rcmpl{b}}$ are
implemented (with \path{op(a, b)} or \path{rop(a, b)} as the name of the implemented
function). Using \path{A} vs. \path{a} implies \path{gr(A)} $\ge$ \path{gr(a)}, where
\path{gr(arg)} returns the grade of the argument \path{arg}:
\begin{itemize}
    \item Inner product, delivering a scalar: $a \cdot b$, \path{dot(a, b)}
    
    \item Outer product (wedge product) of a $j$- and a $k$-vector, delivering a
    $j+k$-vector in $span(a,b)$: $a \wedge b$, \path{wdg(a, b)}
    
    \item Geometric product:  $a \wedgedot b$, \path{operator*(a, b)}
    \begin{itemize}
        \item For vectors $a$ and $b$:  $a \wedgedot b = a \cdot b + a \wedge b$
        \item For a vector $a$ and a multivector $B$:
        $a \wedgedot B = B \gg a + a \wedge B$
                \item For a multivector $A$ and a vector $b$:
        $A \wedgedot b = b \ll A + A \wedge b$
    \end{itemize}

    \item Commutator product (the asymmetric part of the geometric product):
    $\mathrm{cmt(a, b)} \equiv \frac{1}{2}(a \wedgedot b - b \wedgedot a)$, \path{cmt(a,
    b)}
    
    \item Regressive commutator product (the asymmetric part of the regressive geometric
    product): \\
    $\mathrm{rcmt}(a, b) \equiv \frac{1}{2}(\lcmpl{\rcmpl{a} \wedgedot \rcmpl{b} -
    \rcmpl{b} \wedgedot \rcmpl{a}}) = \frac{1}{2}(a \veedot b - b \veedot a)$,
    \path{rcmt}(a, b)
    
    \item Regressive inner product, delivering a pseudoscalar: $a \circ b$, \path{rdot(a,
    b)} (for comparison of the dot product and the regressive dot product for $PGA3DP$ see
    table \ref{tab:pga3dp_dot_products} at the end of section \ref{subsection:basics})
    
    \item Regressive outer product (regressive wedge product): $a \vee b$,
    \path{rwdg(a, b)}
    
    \item Regressive geometric product: $a \veedot b$, \path{rgpr(a, b)}, exclusively for
    $PGA$ 
    
    \item Left (bulk) contraction: $a \lcontr B = a \ll B = \leftbulkdual{a} \vee B $,
    \path{operator<<(a, B)}, which implements \path{l_bulk_contract(a, B)}, exclusively
    for $PGA$
    
    \item Right (bulk) contraction: $B \rcontr a = B \gg a = B \vee \rightbulkdual{a} $,
    \path{operator>>(B, a)}, which implements \path{r_bulk_contract(B, a)},
    exclusively for $PGA$

    \item Left weight contraction: $\leftweightdual{a} \vee B $,
    \path{l_weight_contract(a, B)} 
    
    \item Right weight contraction: $B \vee \rightweightdual{a} $,
    \path{r_weight_contract(B, a)} 

    \item Left bulk expansion: $\leftbulkdual{A} \wedge b $,
    \path{l_bulk_expand(A, b)} 
    
    \item Right bulk expansion: $a \wedge \rightbulkdual{B} $,
    \path{r_bulk_expand(a, B)} 

    \item Left weight expansion: $\leftweightdual{A} \wedge b $,
    \path{l_weight_expand(A, b)} 
    
    \item Right bulk expansion: $a \wedge \rightweightdual{B} $,
    \path{r_weight_expand(a, B)} 
\end{itemize}
where the items from the left contraction downwards are all considered interior products.
The product operation is performed \emph{after} applying a dualization operation to one
operand. \\

All products operate directly on the subspaces occupied by the objects that serve as
arguments for the product operation. \\ 

The product tables for all products are generated in \path{ga/ga_prdxpr}. They can be
derived directly from simple rules that define the products between basis vectors. The
rules are generated for the geometric product, the outer (or wedge) product, the dot
product, the complements and duals of the respective algebra. All resulting product
expressions can be calculated from those few basic rules. Coefficient expressions for
specific argument types like scalars, vector, bivectors, etc. can be generated based on
the product tables using configuration options of \path{ga/ga_prdxpr} for each algebra. \\

$PGA$ explicitly means \emph{Projective Geometric Algebra} in the context of the ga
library. There is also an alternative approach called \emph{Plane-based Geometric Algebra}
(also abbreviated $PGA$) that is based on a space that is dual to the space modelled and
used here. The alternative approach was introduced by~\cite{Gunn_PGA:2020} and further
promoted for applications in projective geometry
by~\cite{Plane-based_PGA_Dorst-DeKennik:2022}. That model uses dual representations of
objects represented in the algebra (it uses $\mathbb{G}^*(3,0,1)$, a space dual to
$\mathbb{G}(3,0,1)$). Planes are used instead of vectors to model everything, e.g. a point
is modelled by three planes intersecting in a point. Both representations are
mathematically equivalent, i.e. they are dual to each other. They are linked by the
definition of complements and the interconnection between geometric and regressive
geometric product, as defined above: $a \veedot b \equiv \lcmpl{\rcmpl{a} \wedgedot
\rcmpl{b}}$. In the plane-based approach the geometric product is used to operate on
complements and duals, which are defined as the primary objects used for the approach,
while the approach introduced by Lengyel in~\cite{Lengyel_pga-illuminated:2024} favors the
use of points, vectors, bivectors and planes plus the explicit use of regressive products
and complements as well as duals for applying geometric operations to these objects. The
latter approach unifies the use of the wedge product for join operations and of the
regressive wedge product for meet operations for application in $EGA$ and $PGA$. \\

\cite{Lengyel_poor-foundations_GA:2024} points out mathematical inconsistencies in
applications of geometric algebra as used in the past quite often by different authors.
Further, hints on mathematical inconsistencies in typical GA approaches can be found as
well in \cite{Kritchevsky_case_against_GA:2024}. The goal of this work is to be as
consistent and mathematically rigorous as possible implementing GA functionality, and at
the same time providing an intuitively applicable framework for modelling physics and
applications. \\
